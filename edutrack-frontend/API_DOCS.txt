URL base

http://localhost:8080 
El frontend NUNCA habla directamente con los microservicios, solo con el gateway. 

Autenticación 
Flujo JWT 

1.  El usuario hace login → recibe un token JWT 
2.  Cada request protegido debe incluir el header: 

Authorization: Bearer <token> 
Estructura del token JWT (payload) 
{ 
  "sub": "email@ejemplo.com", 
  "role": "ROLE_USER", 
  "id": 1, 
  "iat": 1234567890, 
  "exp": 1234567890 
} 
Roles disponibles 

•  USER — usuario normal, puede inscribirse y ver cursos 
•  ADMIN — puede crear/editar/eliminar cursos, categorías y lecciones 

Rutas públicas (sin token) 

•  POST /api/auth/register 
•  POST /api/auth/login 
•  GET /api/courses 
•  GET /api/courses/{id} 
•  GET /api/categories

Flujos principales del frontend 

 
 
 
Flujo de autenticación 
1. POST /api/auth/register → guardar token en localStorage 
2. POST /api/auth/login    → guardar token en localStorage 
3. GET  /api/auth/me       → obtener datos del usuario al cargar la app 
4. Logout                  → eliminar token de localStorage 
Flujo de catálogo 
1. GET /api/categories              → mostrar filtros 
2. GET /api/courses?categoryId=1    → listar cursos filtrados 
3. GET /api/courses/{id}            → ver detalle con lecciones 
Flujo de inscripción y progreso 
1. POST /api/enrollments              → inscribirse 
2. GET  /api/enrollments/progress/{courseId} → ver progreso actual 
3. POST /api/enrollments/progress     → marcar lección completada 
4. GET  /api/enrollments/my           → listar mis cursos 
Flujo de admin 
1. POST /api/categories               → crear categoría 
2. POST /api/courses                  → crear curso 
3. POST /api/courses/{id}/image       → subir portada a S3 
4. POST /api/courses/{id}/lessons     → agregar lecciones 
5. PUT  /api/courses/{id}             → editar curso 
6. DELETE /api/courses/{id}           → eliminar curso 

Notas importantes para el frontend 

1.  Token storage — guardar en localStorage con key token 
2.  Header de autorización — siempre Bearer <token> (con espacio) 
3.  Subida de imágenes — usar multipart/form-data, NO application/json 
4.  Paginación — los listados devuelven objeto con content[], totalPages, 

totalElements 

5.  Roles — el campo role en la respuesta de login es "USER" o "ADMIN" (sin prefijo 

ROLE_) 

6.  CORS — el gateway permite localhost:3000, localhost:5173 y localhost:5174 
7.  Progreso — si el usuario no está inscrito, GET progress devuelve 400 (manejar con 

catch) 

8.  imageUrl — puede ser null si el curso no tiene portada, manejar con fallback
